# win-agent 从 0 到 1 项目支持度分析与优化计划

## 一、当前支持度评估

### 现状总结：支持度低，核心流程多处断裂

win-agent 当前的设计假设是**在已有代码库上进行迭代开发**。对于从零开始的项目，以下环节存在明确问题：

### 1.1 Init 阶段 —— 空目录被跳过

**位置**: `src/cli/init.ts:343`

```typescript
if (!detectExistingCode(workspace)) {
  console.log('   空目录，跳过');
}
```

空目录时，Phase 7（工作空间分析）被完全跳过：
- `overview.md` 不生成
- `development.md` 不生成（退化为 TODO 骨架）
- `validation.md` 不生成（退化为 TODO 骨架）

**后果**: DEV 在 Phase 3 中依赖这三个文件指导开发，空骨架意味着 DEV 不知道用什么命令安装依赖、构建、测试、验收。

### 1.2 PM 角色 —— 缺少 0-to-1 专属流程

**位置**: `src/templates/roles/PM.md`

PM 的"需求处理"流程假设：
- 项目技术栈已确定（PM 不做技术选型）
- 项目结构已存在（PM 让 DEV 修改/新增模块）
- 验收手段已就绪（PM 按 validation.md 审核）

**0-to-1 场景缺失**:
- 无**技术选型**讨论流程（用什么框架、什么数据库、什么部署方式）
- 无**项目脚手架搭建**阶段（第一个 task 应该是 `npx create-xxx` 或手动搭建）
- 无**基础架构确认**环节（目录结构、模块划分、关键技术决策）
- PM 的 Spec 格式（用户故事 + 验收标准）适合功能迭代，但 0-to-1 的第一批 task 往往是"搭建项目骨架"、"配置 CI/CD"等基础设施任务

### 1.3 DEV 角色 —— 5 步流程对空项目不适用

**位置**: `src/templates/roles/DEV.md` Phase 3

| Step | 对已有项目 | 对空项目 |
|------|-----------|---------|
| Step 1 环境准备 | 读 development.md 执行安装命令 | development.md 是空骨架，无命令可执行 |
| Step 2 基线验证 | 跑 lint/build/test 记录当前状态 | 没有代码，没有配置文件，无法跑任何检查 |
| Step 3 编码实现 | 在已有代码上修改/新增 | 需要先创建项目（脚手架/手动搭建） |
| Step 4 代码检查 | 跑 lint/build/test 确保不引入新问题 | 如果 Step 3 包含了脚手架搭建，这些命令可能需要先在 development.md 中注册 |
| Step 5 E2E 验收 | 按 validation.md 执行 | validation.md 是空骨架 |

**核心矛盾**: 0-to-1 的第一个 task 需要同时创建代码 + 创建开发规范文档，但当前流程假设文档先于代码存在。

### 1.4 文档生命周期 —— 缺少动态更新机制

init 时生成的 development.md / validation.md 是**一次性快照**。但 0-to-1 项目中：
- 脚手架搭建后才能确定具体的 lint/build/test 命令
- 随着项目演进，命令和规范会变化
- 当前没有机制让 DEV 在搭建完脚手架后更新这些文档

### 1.5 知识库 —— 冷启动问题

对已有项目，init 时可以导入参考资料、扫描代码生成知识。
对空项目，知识库完全为空，PM 和 DEV 缺少上下文。

## 二、优化方案

### 2.1 Init 流程增加 0-to-1 模式

**改动文件**: `src/cli/init.ts`

在 Phase 4（项目信息）之后，增加项目模式检测/选择：

```
4️⃣  项目信息
    项目名称: my-app
    项目描述: 一个 xxx 应用

4.5️⃣ 项目模式
    检测到空目录。
    ? 选择项目模式:
      > 从零开始（0-to-1）—— 我要创建一个新项目
        已有代码 —— 我会把代码放进来
```

选择"从零开始"后：
1. 在 DB 中写入 `project_config { key: 'project_mode', value: 'greenfield' }`
2. 收集额外信息（可选，PM 后续也会引导）：
   - 项目类型（Web 前端 / 全栈 / API 后端 / CLI 工具 / 库）
   - 偏好技术栈（如有）
   - 部署目标（如有）
3. 将收集到的信息写入知识库（category='requirement'）
4. 跳过 Phase 7（工作空间分析），但生成 **0-to-1 专用的 docs 模板**（而非空骨架）

**0-to-1 专用 docs 模板**:

`development.md`:
```markdown
# 开发指南

_项目尚未创建，本文件将在脚手架搭建完成后由 DEV 自动更新。_

## 环境准备
待脚手架任务完成后补充。

## 开发命令
待脚手架任务完成后补充。

## 编码要求
待脚手架任务完成后补充。
```

`validation.md`:
```markdown
# 验收规范

_项目尚未创建，本文件将在脚手架搭建完成后由 DEV 自动更新。_

## 代码检查
待脚手架任务完成后补充。

## E2E 验收
待脚手架任务完成后补充。
```

### 2.2 PM 角色增加 0-to-1 流程

**改动文件**: `src/templates/roles/PM.md`

在"需求处理"之前增加 **项目启动流程**，仅在 greenfield 模式 + 无已完成 task 时触发：

#### 新增：项目启动流程

```markdown
## 项目启动（仅 0-to-1 项目首次触发）

**触发条件**: project_config 表中 project_mode='greenfield' 且 tasks 表无 status='done' 的记录。

### Step 1 — 需求探索
与用户深入讨论：
1. 核心问题：这个项目要解决什么问题？用户是谁？
2. MVP 边界：第一个可交付版本应该包含什么？明确不包含什么？
3. 参考系统：有没有类似的产品/项目可以参考？

### Step 2 — 技术选型讨论
基于需求和用户已声明的约束/偏好，提出技术选型建议：
1. 语言 & 框架（给出 2-3 个选项及 trade-off）
2. 数据存储方案
3. 部署方式
4. 关键第三方依赖
**必须等用户确认后才进入下一步。**

### Step 3 — 架构规划
确认选型后，输出初步架构规划：
1. 目录结构草案
2. 核心模块划分
3. 数据模型概要（如适用）
4. API 路由概要（如适用）
将规划写入 `.win-agent/docs/spec/architecture.md`，写入知识库（category='convention'）。

### Step 4 — 脚手架任务派发
创建第一个特殊 task —— **脚手架搭建**（tag: scaffold）：
- directive 中包含完整的技术选型决策、架构规划、目录结构要求
- 验收标准：
  1. 项目可成功安装依赖
  2. 可成功构建（build 通过）
  3. 开发服务器可启动（如适用）
  4. Lint 配置就绪且通过
  5. DEV 已更新 development.md 和 validation.md

### Step 5 — 回到常规流程
脚手架完成后，project_mode 保持 greenfield 但已有 done task，
后续需求按常规"需求处理"流程进行。
```

### 2.3 DEV 角色增加脚手架开发流程

**改动文件**: `src/templates/roles/DEV.md`

在 Phase 3 开头增加脚手架分支：

```markdown
### Phase 3 — 开发和自测

**脚手架任务（directive 中含 tag: scaffold）特殊流程：**

> 脚手架任务不执行常规 Step 1-2（无已有环境和基线），改为以下流程：

| Step | 做什么 |
|------|--------|
| **S1 — 创建项目** | 根据 directive 中的技术选型，使用脚手架工具（如 `npx create-xxx`）或手动搭建项目结构 |
| **S2 — 基础配置** | 配置 lint/format/test 工具、tsconfig、.gitignore 等开发基础设施 |
| **S3 — 验证可用** | 安装依赖 → build 通过 → dev server 可启动（如适用）→ lint 通过 |
| **S4 — 更新 docs** | **必须**根据实际项目配置更新 `.win-agent/docs/development.md` 和 `.win-agent/docs/validation.md`（从空模板填充为实际命令） |
| **S5 — 提交验收** | git init（如需）+ git add + commit，进入 Phase 4 |

**常规任务（非脚手架）按原有 Step 1→2→3→4→5 执行。**
```

### 2.4 DEV 可更新 docs 规则文件的权限放开

**改动文件**: `src/templates/roles/DEV.md`

当前 DEV 被禁止操作 `.win-agent/` 目录（`.win-agent/docs/` 除外但仅限经验归档文件）。需要明确允许 DEV 在脚手架任务中更新 `development.md` 和 `validation.md`。

修改规则为：
```
禁止操作 `.win-agent/` 目录，以下例外：
- `.win-agent/docs/known-issues.md` / `dev-notes.md` / `efficiency-and-skills.md`（经验归档，任何时候可写）
- `.win-agent/docs/development.md` / `validation.md`（仅在脚手架任务中可更新）
```

### 2.5 Init 后 overview.md 的延迟生成

**改动文件**: `src/cli/init.ts`, `src/engine/scheduler.ts`（或 `auto-trigger.ts`）

脚手架 task 完成后，engine 自动触发一次 overview.md 生成：

1. `auto-trigger.ts` 检测到脚手架 task 状态变为 `done`
2. 触发 PM 执行工作空间分析 prompt（复用 `buildWorkspaceAnalysisPrompt`）
3. 写入 `overview.md`

这样 overview.md 能准确反映实际创建的项目结构，而不是在空目录上强行分析。

### 2.6 development.md / validation.md 的演进式更新

**新增机制**: 当 DEV 在脚手架任务中更新了 docs 文件后，后续 init 重新运行时应保留 DEV 的修改而非覆盖。

当前 `ensureDocsFiles()` 已经有 `if (!fs.existsSync(filePath))` 的保护，所以不会覆盖已有文件。但需要确保：
- `update` 命令（更新角色模板）不会覆盖 docs 文件 ✓（当前只更新 roles）
- re-init 时如果用户选择重新运行，Phase 7 不会覆盖 DEV 已更新的 docs
  - **建议**: re-init 时检测到 docs 文件非骨架（无 TODO 标记 / 非空占位），提示用户确认是否覆盖

## 三、实施优先级

| 优先级 | 改动 | 工作量 | 影响 |
|--------|------|--------|------|
| **P0** | 2.2 PM 增加项目启动流程 | 中 | 核心：没有这个，0-to-1 完全无法启动 |
| **P0** | 2.3 DEV 增加脚手架开发流程 | 中 | 核心：DEV 需要知道如何处理"创建项目"这个特殊 task |
| **P1** | 2.1 Init 增加 0-to-1 模式 | 中 | 改善体验：让用户在 init 阶段就声明意图 |
| **P1** | 2.4 DEV docs 写权限放开 | 小 | 配合 2.3，让 DEV 能更新规范文件 |
| **P2** | 2.5 overview.md 延迟生成 | 中 | 改善质量：确保概览文档准确 |
| **P2** | 2.6 docs 演进式更新保护 | 小 | 防御性：防止 re-init 覆盖 DEV 的工作 |

## 四、可能的架构风险

1. **脚手架 task 失败的回滚**: 脚手架 task 失败后，workspace 可能处于半成品状态（部分文件已创建）。DEV 的 `git revert` 策略对此可能不够——因为可能还没有 git 仓库。需要考虑 `rm -rf` 回滚或分步提交。

2. **PM 技术选型能力边界**: PM 本质上是 LLM，技术选型建议可能不准确。需要强调"必须等用户确认"，PM 不能自行决定技术栈。

3. **docs 更新的时序**: 脚手架 task 中 DEV 更新 docs 后，如果后续 task 的 directive 是在 docs 更新前由 PM 编写的，PM 可能引用了过时的 docs 内容。但由于 DEV 每次唤醒都会重新读 docs 文件，这个问题不大。

4. **多次 0-to-1**: 用户可能在同一个 workspace 下先搭建后端再搭建前端（monorepo 逐步构建）。当前方案的"脚手架 task"是一次性的，需要考虑是否支持多次脚手架任务。建议：不限制 scaffold tag 的使用次数，PM 可以在任何时候创建新的脚手架任务。

## 五、未来展望

- **项目模板库**: 内置常见项目模板（Next.js / Express+Prisma / FastAPI 等），用户选择后自动生成脚手架 directive，减少 PM/DEV 的试错
- **技术选型知识库**: 预置常见技术栈的优劣对比，辅助 PM 给出更好的建议
- **渐进式 Monorepo 支持**: 支持"先创建 backend，再加 frontend"的渐进式项目构建
- **CI/CD 脚手架**: 0-to-1 阶段自动建议并配置 CI/CD pipeline
