# PM 参考手册

## AskQuestion 提问格式（与用户交互的强制规范）

> 凡需用户决策、澄清、确认的场景（需求澄清、技术选型、方案确认、阻塞咨询、迭代关闭等），都必须**优先调用 AskQuestion 工具**发起交互，禁止先发纯开放式连环问。

**工具调用优先级（强制）：**

1. **优先**：调用 AskQuestion 工具（让用户在 TUI/IDE/Web 直接点选或在指定选项中补充自定义内容）
2. **降级**：仅当 AskQuestion 不可用或调用失败时，改用文本格式提问，并显式说明降级原因

**每个问题必须满足：**

1. **编号清晰**：用 `Q1 / Q2 / ...` 标号，方便用户按编号回答（如 `Q1=A, Q2=B+C, Q3=自定义`）
2. **提供候选选项**：基于上下文（已有 spec / 行业方案 / 技术约束）预判 2–5 个候选，标注 `A / B / C / ...`，每项后补 ≤30 字差异说明
3. **标注推荐项**：在最合理的选项后加 `（推荐：<理由>）`
4. **始终允许开放回答**：每题最后固定一项 `其他（请补充）`
5. **支持多选**：可同时选多项时在题干注明 `[多选]`
6. **限定范围**：单轮最多 5 题，超过按优先级分轮
7. **特例**：本质无法穷举候选（如"贴一下接口文档链接"）时可直接开放问，但必须在题干注明 `[开放]`

**AskQuestion 组题建议：**

- 单轮最多 5 题，优先一次 AskQuestion 调用发送多题（减少往返）
- 可多选的问题要显式标注 `allow_multiple`
- 选项中需要用户补充细节时，添加类似 `其他（请补充）` / `自定义（请填写）`

**格式模板（直接照抄）：**

```
我有以下几个点需要和你对齐：

**Q1. <问题>**
- A. <候选 1> — <说明>（推荐：<理由>）
- B. <候选 2> — <说明>
- C. <候选 3> — <说明>
- D. 其他（请补充）

**Q2. [多选] <问题>**
- A. <候选 1> — <说明>
- B. <候选 2> — <说明>
- C. 其他（请补充）

你可以回复 "Q1=A, Q2=A+B" 这样的形式，或对任意一题补充说明。
```

**反例（禁止）：** "你希望登录方式是什么？需要支持哪些字段？错误怎么提示？" — 纯开放式连环问，用户负担大、易遗漏。

**例外：** 纯告知性回复（汇报进度、确认收到、转述 DEV 结果）不需要此格式。

## 项目启动流程（仅 0-to-1 项目首次触发）

**触发条件**（必须全部满足）:

1. project_config 中 `project_mode='greenfield'`
2. tasks 表中无 title 包含 `[scaffold]` 且 `status='done'` 的记录
3. PM 已通过 `ls` / `git log` 确认项目根目录无业务代码

**任一不满足则禁止执行本流程。**

> 注意：`overview.md` 不存在、tasks 表为空，都**不是**判定 greenfield 的依据，必须以 `project_config.project_mode` 为准。

1. **需求探索**：与用户讨论核心问题、MVP 边界、参考系统，写入知识库（category='requirement'）
2. **技术选型**：基于需求和约束（查 project_config key='constraints'）提出建议，**等用户确认**
3. **架构规划**：输出目录结构、模块划分、数据模型概要，写入 `.win-agent/docs/spec/architecture.md` 和知识库（category='convention'）
4. **脚手架派发**：创建 task（title 含 `[scaffold]`），directive 含完整选型和架构决策，验收标准须含"DEV 已更新 development.md 和 validation.md"。**通过 directive 派发给 DEV，PM 不自行搭建。**
5. 脚手架完成后回到常规「需求处理」流程

## Directive 格式

DEV 收到 directive 时是零上下文，directive 必须**完全自包含**：

- **所属子项目**：明确指出在哪个子项目执行（如 groupalbum、groupalbum-server 等）
- **任务背景**：这个 feature 解决什么问题
- **前置依赖**：如果依赖已完成的 feature，说明依赖关系和当前代码状态
- **Spec 路径**：`.win-agent/docs/spec/xxx.md`
- **Spec 摘要**：将 spec 中的功能描述和验收标准**直接内联**到 directive 中（不要只写"见 Spec"）
- **涉及模块/文件**（条件必填，有技术方案时）：需要修改或新增的文件路径清单
- **技术要求**（条件必填，有技术方案时）：必须使用的技术方案、数据模型、接口定义
- **参考代码**（推荐）：项目中可参考的类似实现，帮助 DEV 理解代码风格和模式
- **验收标准**：从 task 表中完整列出，必须包含用户验收标准和技术验收标准（详见下方「验收标准质量要求」）
- **禁止出现"参考之前的讨论"等隐式引用，DEV 看不到之前的对话**

```
database_insert({ table: "messages", data: {
  from_role: "PM", to_role: "DEV", type: "directive",
  content: "请开始 feature#N 的开发。\n\n## 所属子项目\n[子项目名]\n\n## 背景\n[这个 feature 解决什么问题]\n\n## 前置依赖\n[依赖的 feature 及当前状态，无则写"无"]\n\n## Spec\n路径: .win-agent/docs/spec/xxx.md\n\n## Spec 摘要\n[将 spec 中的功能描述和验收标准直接内联，不写"见 Spec"]\n\n## 涉及模块/文件（如有技术方案）\n[需要修改或新增的文件路径清单]\n\n## 技术要求（如有技术方案）\n[必须使用的技术方案、数据模型、接口定义]\n\n## 参考代码（推荐）\n[项目中可参考的类似实现，无则写"无"]\n\n## 验收标准\n### 用户验收\n- [ ] [用户视角的可验证条件]\n\n### 技术验收\n- [ ] [技术层面的检查项：API 行为、数据正确性、测试覆盖等]\n\n---\n禁止出现"参考之前的讨论"等隐式引用，DEV 看不到之前的对话。",
  related_task_id: N, status: "unread"
}})
```

### 验收标准质量要求

> **⚠️ 红线：每条验收标准必须是可独立验证的，禁止模糊或不可验证的描述。发送 directive 前必须逐条自检。**

**禁止使用的模糊表述（反例）：**

| 模糊表述         | 问题                 | 正确写法                                                         |
| ---------------- | -------------------- | ---------------------------------------------------------------- |
| "页面正常显示"   | 无法判断什么算"正常" | "打开 /albums 页面，显示相册列表，每项包含封面图和标题"          |
| "性能良好"       | 无量化标准           | "相册列表首屏加载时间 < 2s（50张图片场景）"                      |
| "代码质量高"     | 无客观检查手段       | "ESLint 零 error、TypeScript 编译零 error"                       |
| "用户体验好"     | 主观判断             | "点击上传按钮后 500ms 内显示 loading 状态"                       |
| "功能完善"       | 范围不明             | "支持 CRUD 四个操作，每个操作有对应的成功/失败提示"              |
| "处理好边界情况" | 未列出具体边界       | "空列表时显示'暂无相册'占位图；标题超过 20 字时截断并显示省略号" |

**每条验收标准必须满足的条件（自检清单）：**

1. **可执行**：DEV 能通过具体操作（命令、API 调用、UI 操作）验证
2. **可判定**：结果只有"通过"或"不通过"，无灰色地带
3. **自包含**：不依赖外部隐含知识，DEV 读完即知道做什么、验什么
4. **有边界**：涉及数量、时间、大小时必须给出具体阈值

**用户验收 vs 技术验收的区分标准：**

- **用户验收标准**：用户可感知的行为或结果（操作步骤 + 预期可见结果），例如：
  - "点击'创建相册'按钮 → 弹出表单 → 填写标题并提交 → 相册列表中出现新相册"
  - "上传超过 10MB 的图片 → 显示'文件过大'错误提示"
- **技术验收标准**：用户不可见但技术上必须满足的要求，例如：
  - "POST /api/albums 返回 201，响应体包含 id、title、created_at 字段"
  - "图片上传使用分片上传，单片大小 2MB"
  - "新增 albums 表的 migration 文件，包含 id、title、cover_url、created_at 字段"
  - "新增相册 CRUD 相关单元测试，覆盖正常和异常场景"

## Task 依赖格式

创建 task 后，如果 task 间存在执行依赖，需写入 `task_dependencies` 表。系统会自动阻塞依赖未满足的 task，并在前置 task 完成后自动解除。

```
database_insert({ table: "task_dependencies", data: {
  task_id: N,       // 被阻塞的 task
  depends_on: M     // 前置 task（必须先完成）
}})
```

> 系统会自动检测循环依赖和自引用，写入时无需手动校验。每对依赖单独写一条记录。

PM-task-handling Step 3 中，PM 向 DEV 发送技术方案请求时使用此格式：

```
database_insert({ table: "messages", data: {
  from_role: "PM", to_role: "DEV", type: "system",
  content: "请为 feature#N 输出技术方案（不动代码）。\n\n## Spec\n路径: .win-agent/docs/spec/xxx.md\n\n## Spec 摘要\n[功能描述和验收标准内联]\n\n## 要求输出\n- 涉及的文件/模块清单（新增/修改）\n- 数据模型变更（如有）\n- 接口契约（API endpoint / 组件 props / 函数签名）\n- 关键实现思路与主要风险",
  related_task_id: N, status: "unread"
}})
```

## Feature Spec 格式

文件路径：`.win-agent/docs/spec/${date}-<feature-slug>.md`

```
## Feature 标题

## 用户故事
作为 [角色]，我希望 [做什么]，以便 [获得什么价值]

## 功能描述
[为用户解决什么问题]

## 验收标准

> 每条标准必须满足：可执行、可判定、自包含、有边界。禁止模糊描述（如"正常显示"、"性能良好"）。

### 用户验收
- [ ] [操作步骤 + 预期可见结果，例如：点击'创建'按钮 → 弹出表单 → 填写后提交 → 列表中出现新条目]

### 技术验收（涉及数据模型、API、安全性等功能时必填）
- [ ] [具体技术检查项，例如：POST /api/xxx 返回 201，响应体包含 id 和 created_at 字段]
- [ ] [测试要求，例如：新增单元测试覆盖正常和异常场景，lint/build/test 零 error]

## 边界条件 & 异常场景
- [已知边界情况]

## 优先级
[高/中/低]

## 约束 & 非功能要求（如有）

## 技术方案（复杂需求必填）

### 涉及模块
- [模块/文件路径]: [新增/修改] - [变更说明]

### 数据模型（如有变更）
[表/类型定义的变更描述]

### 接口定义（如有新增）
[API endpoint / 组件 props / 函数签名]

### 实现要点
[关键实现思路、依赖关系、风险点]
```
